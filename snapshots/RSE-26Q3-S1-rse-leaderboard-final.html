Methodology

Four signals · two attributions

Scope injection count — items added to the sprint after planning, owned by this RSE
Two sub-buckets, mirroring the existing scope-composition framework:
• Day-1 dump: created within 24h after sprint.startDate.
• True mid-sprint injection: created more than 24h after sprint start.
Close attribution: the RSE is whoever holds fields.assignee.displayName at snapshot time.
Event-time attribution: without changelog wiring this falls back to current assignee, so close and event columns are identical this sprint.
Mid-sprint churn events — re-points and scope edits while the sprint is open
Counts changelog events between sprint.startDate and sprint.endDate on these fields: Story Points (customfield_10026), summary, description, Sprint (customfield_10020 — pulled in or pushed out mid-sprint).
Implementation note: the finalize run does not yet pull changelog (searchJiraIssuesUsingJql doesn't expose expand). Adding it will fill this column on a future finalize. Until then it shows —.
Carry-over rate — share of committed work this RSE didn't close
(SP owned at sprint close that didn't reach a Done status) / (SP owned at sprint close).
Locked this snapshot: every issue in RSE 26Q3 S1 reached a Done status category by close, so carry-over is 0% for all 20 RSEs.
Close attribution only — at-event timing isn't meaningful for an end-of-sprint snapshot.
Tagging coverage — % of this RSE's owned injections with Contamination Source set
Numerator: this RSE's day-1 + true injections where customfield_10313.value is non-null.
Denominator: this RSE's total day-1 + true injections.
N/A if denominator is zero (RSE owned no injections). Team-wide: 33 of 58 owned injections tagged (57%).
Composite volatility score — single number, 0–100, lower is healthier
Each component is a 0–1 ratio, multiplied by its weight, then summed. Weights:
  • True mid-sprint injection density  ×30
  • Carry-over rate  ×25
  • Untagged share of owned injections  ×20
  • Churn density (re-points + scope edits per owned item)  ×15
  • Day-1 dump density  ×10
Renormalization: churn is null this sprint (changelog not wired), so its 15-point weight redistributes proportionally across the other four components.
Density denominators use owned-item count, so one injection on an RSE carrying 5 items is worse than the same injection on someone carrying 11.
Bands: Clean (<10) · Healthy (10–25) · Watch (25–45) · Volatile (45–70) · Critical (≥70).
If you'd rather see a "health score": use 100 − volatility. Same information, inverted direction.

Why dual attribution matters

When the close-attribution and event-time columns differ for an RSE, a ticket has been reassigned mid-sprint. Investigating the divergence is often more useful than the absolute numbers: it surfaces hand-offs (who took it from whom), parking lots (an RSE who keeps inheriting other people's churn), and ownership churn (a ticket that bounced through three people). This sprint's columns are identical because event-time attribution falls back to close attribution without changelog wiring.
