Methodology

Four signals · two attributions

Scope injection count — items added to the sprint after planning, owned by this RSE
Two sub-buckets, mirroring the existing scope-composition framework:
• Day-1 dump: created within 24h after sprint.startDate.
• True mid-sprint injection: created more than 24h after sprint start.
Close attribution: the RSE is whoever holds fields.assignee.displayName at snapshot time.
Event-time attribution: the RSE is whoever was first assigned to the issue (from changelog, field assignee, earliest to value; falls back to current assignee if never reassigned). Divergence = a ticket changed hands after creation.
Mid-sprint churn events — re-points and scope edits while the sprint is open
Counts changelog events between sprint.startDate and min(sprint.endDate, now) on these fields: Story Points (customfield_10026), summary, description, Sprint (customfield_10020 — pulled in or pushed out mid-sprint).
Close attribution: the event counts toward the current assignee.
Event-time attribution: the event counts toward whoever held the assignee at the moment of the change.
Implementation note: the daily run does not yet pull changelog. Adding expand: ["changelog"] to the step-0 JQL will fill this column. Until then it shows —.
Carry-over rate — share of committed work this RSE didn't close
Defined at finalize only: (SP owned at sprint close that didn't reach a Done status) / (SP owned at sprint close).
Mid-sprint, the column shows —. The denominator includes day-1 dump and true injections so RSEs aren't penalized for items they never agreed to take.
Close attribution only — at-event timing isn't meaningful for an end-of-sprint snapshot.
Tagging coverage — % of this RSE's owned injections with Contamination Source set
Numerator: this RSE's day-1 + true injections where customfield_10313.value is non-null.
Denominator: this RSE's total day-1 + true injections.
N/A if denominator is zero (RSE owned no injections). The team-wide tagging coverage figure on the main snapshot is the weighted aggregate of these per-RSE rows.
Composite volatility score — single number, 0–100, lower is healthier
Each component is a 0–1 ratio, multiplied by its weight, then summed. Weights:
  • True mid-sprint injection density  ×30
  • Carry-over rate  ×25
  • Untagged share of owned injections  ×20
  • Churn density (re-points + scope edits per owned item)  ×15
  • Day-1 dump density  ×10
Renormalization: when a component isn't measurable yet (carry-over before close, churn before changelog wiring), its weight is redistributed proportionally across the rest. This keeps the score on the same 0–100 scale all sprint long instead of starting low and ramping up artificially.
Density denominators use owned-item count, so one injection on an RSE carrying 4 items is worse than the same injection on someone carrying 12.
Bands: Clean (<10) · Healthy (10–25) · Watch (25–45) · Volatile (45–70) · Critical (≥70).
If you'd rather see a "health score": use 100 − volatility. Same information, inverted direction.

Why dual attribution matters

When the close-attribution and event-time columns differ for an RSE, a ticket has been reassigned mid-sprint. Investigating the divergence is often more useful than the absolute numbers: it surfaces hand-offs (who took it from whom), parking lots (an RSE who keeps inheriting other people's churn), and ownership churn (a ticket that bounced through three people).
